# F6 Audit — SwarmViz + LivingSwarmGrid

**Date:** 2026-04-22 (late session)
**Scope:** The two god-viz files flagged in the 2026-04-22 dashboard UI/UX audit (finding F6). Read-only pass; no code changes. Produces a structured decomposition plan so a future F6 session lands refactors cleanly.

**Method:** structural read of both files + sampling of the ~15 helper modules already extracted. Claims grounded in concrete file:line references.

## Summary

| File | Lines | Inline styles | Top-level shape |
|---|---|---|---|
| [`SwarmViz.tsx`](../src/cli/dashboard/src/components/viz/SwarmViz.tsx) | 1513 | 34 | Composition root for the VIZ tab. Owns shared playhead + grid-mode + chronicle-filter + focused-side state. Delegates rendering to 8 child components. |
| [`grid/LivingSwarmGrid.tsx`](../src/cli/dashboard/src/components/viz/grid/LivingSwarmGrid.tsx) | 1243 | 27 | Per-leader canvas coordinator. Owns canvas element + ResizeObserver + hover/click/popover state. Delegates drawing to 6 layer modules. |

**Neither file is a classic "god file."** Both already delegate heavily to helpers. The 1500 / 1243 line counts come from (a) dense hook setup, (b) long render effects with canvas draw choreography, and (c) pervasive inline styling for DOM chrome around the canvas.

The 2700 combined lines are organized; they're not spaghetti. What they lack is a cleanly-extracted state machine + a separated render orchestrator. That's what F6 should deliver.

## SwarmViz.tsx findings

### What it owns

From [lines 68-380](../src/cli/dashboard/src/components/viz/SwarmViz.tsx#L68-L380) the composition root manages:

1. **Playback state** — `currentTurn`, `playing`, `speed` + timer effect (~40 lines)
2. **Grid mode** — `gridMode` with localStorage persistence (~15 lines)
3. **Grid settings** — full `GridSettings` object with schema-versioned localStorage blob (`SETTINGS_VERSION = 3` guard, ~25 lines)
4. **Crisis toast** effect scanning both event streams for un-toasted crises with sessionStorage dedup (~80 lines, one of the densest blocks)
5. **Search + chronicle filter + focused-side + palette + hovered agent IDs** (~30 lines of useState)
6. **Timelapse recording** (`useVizSnapshots`) + mode-driven layout tweaks
7. **Jump-to-reports** navigation handler
8. **Sound cues** hook wiring

### Issues

**S1 — Crisis toast effect is a hook candidate.** [Lines 234-330ish](../src/cli/dashboard/src/components/viz/SwarmViz.tsx#L234) — an 80-line useEffect that scans events, dedups via sessionStorage, formats a toast payload, and fires it. Belongs in `useCrisisToast(state, gridSettings.alerts)`. Testable via pure `findLatestUntoastedCrisis(events, seen)` helper (matches the pattern `useGameState.helpers` / `useLoadPreview.helpers` established earlier).

**S2 — Grid settings localStorage versioning is a hook candidate.** [Lines 199-232](../src/cli/dashboard/src/components/viz/SwarmViz.tsx#L199-L232) — versioned-blob pattern identical to `useGamePersistence` but with a different `SETTINGS_VERSION` constant. Extract `useGridSettings()` co-located with `GridSettingsDrawer`. Shares the same schema-bumping discipline; when `GridSettingsDrawer` adds a new boolean toggle default-off, `SETTINGS_VERSION` bumps and stored `true` doesn't override.

**S3 — `Kbd` helper is trivial but in the wrong place.** [Lines 29-48](../src/cli/dashboard/src/components/viz/SwarmViz.tsx#L29-L48) — small keyboard-chip component used once in the footer legend. Either inline into the footer or move to `shared/Kbd.tsx`. Occupying module-top slot on the god-viz file is noise.

**S4 — 34 inline styles.** Same F4 pattern as prior dashboard batches. High-volume, low-risk migration to `SwarmViz.module.scss`. Per-leader colors already flow through CSS custom properties on LivingSwarmGrid; SwarmViz's chrome is simpler (fixed palettes, no dynamic per-row colors).

**S5 — `setGridModeState` + `setGridMode` dual-state pattern.** [Lines 84-96](../src/cli/dashboard/src/components/viz/SwarmViz.tsx#L84-L96) — private useState wrapped by a callback that persists to localStorage. Fine, but the same shape as the grid-settings pattern; both could share a `useLocalStorageState<T>(key, defaultValue, version?)` utility. Low-pri refactor.

**S6 — Chronicle hover payload typing is duplicated.** Both SwarmViz and LivingSwarmGrid define a `{ kind, side, turn }` shape inline. Export the type from `EventChronicle.tsx` (where the source of truth lives) + import on both sides.

**S7 — Prop drilling depth.** SwarmViz passes ~20 props to each LivingSwarmGrid instance. When F2+F3 arena lands (N-leader), this becomes N-card prop threading. Worth thinking about context-based dispatch or a `<GridSlot />` sub-component wrapper that pulls the per-slot state from a `GridContext`.

### Already good

- Delegation is clean: 8 child components + 5 hooks + helpers pulled out of line
- `useVizSnapshots` abstracts snapshot generation
- Media query wrapper (`useMediaQuery`) used consistently
- GridMode + ClusterMode union types are well-designed

## LivingSwarmGrid.tsx findings

### What it owns

From [lines 134-200+](../src/cli/dashboard/src/components/viz/grid/LivingSwarmGrid.tsx#L134) the canvas coordinator manages:

1. **Refs** — container, canvas wrap, overlay canvas, ResizeObserver cleanup ([lines 169-250](../src/cli/dashboard/src/components/viz/grid/LivingSwarmGrid.tsx#L169-L250))
2. **Canvas sizing** — DPR-aware resize effect
3. **State** — `size`, `hovered`, `cursor`, `popover`, `rosterOpen`, `hoveredTile`, `revealed` (7 useState hooks)
4. **Grid state memos** — `positions`, `deptCentersOverlay`, `modeClusterMode`, `moodTintedSideColor` (4 useMemo)
5. **Game-of-Life ambient overlay** — 5 refs (`golStateRef`, `lastGolTurnRef`, `lastGolModeRef`, `lastGolFilterRef`, `lastFlareSignatureRef`) + ~60-line seed+warmup block inside the main render effect
6. **Main render effect** — [lines 351-750+](../src/cli/dashboard/src/components/viz/grid/LivingSwarmGrid.tsx#L351) — the biggest single block. Orchestrates GoL → GlyphLayer → FlareLayer → ModeOverlayLayer → HudLayer → crosshair + labels. ~400 lines of canvas draw choreography.
7. **Pointer + click handlers** — inline `onMouseMove`, `onMouseLeave`, `onClick`, `onTouchStart` with hit-testing against both glyphs and GoL tiles (~150 lines)
8. **JSX return** — container + canvas + buttons + RosterDrawer + FeaturedSpotlight + ClickPopover + DivergenceEmptyState + TileTooltip ([lines 800-1243](../src/cli/dashboard/src/components/viz/grid/LivingSwarmGrid.tsx#L800))

### Issues

**L1 — Main render effect should be extracted to `renderFrame()`.** [Lines 351-750ish](../src/cli/dashboard/src/components/viz/grid/LivingSwarmGrid.tsx#L351) — a ~400-line useEffect calling `ctx.clearRect` → `drawGol` → `drawGlyphs` → `drawFlares` → `drawForgeHeatmap` / `drawEcologyResourceMap` / `drawDivergenceHighlight` → `drawHud` → crosshair + label drawing. All draw calls are already pure (the layer modules are testable in isolation). The orchestration itself — "in what order, with what alpha, under what conditions" — is the piece not extracted.

Proposed: `renderFrame({ ctx, state, gol, positions, flares, mode, ... })` as a pure function in a new `renderFrame.ts`. The useEffect becomes a 10-line wrapper that collects args + calls it. Immediately gives you a testable render (you can pass a fake ctx that records draw calls + assert order / args).

**L2 — Pointer handlers belong in a `useCanvasInteractions` hook.** [Lines 530-680ish](../src/cli/dashboard/src/components/viz/grid/LivingSwarmGrid.tsx#L530) — `onMouseMove` does: cursor position calc → glyph hit-test → tile hit-test → state updates for hovered / cursor / hoveredTile. `onClick` does similar for popovers. `onTouchStart` normalizes touch to mouse semantics.

Extract to `useCanvasInteractions({ positions, cells, onHoverChange, ... })` returning `{ onMouseMove, onMouseLeave, onClick, onTouchStart, hovered, cursor, hoveredTile, popover }`. Single responsibility, testable with synthetic event objects.

**L3 — 27 inline styles concentrated in JSX return.** [Lines 800-1243](../src/cli/dashboard/src/components/viz/grid/LivingSwarmGrid.tsx#L800) — container div with media-query-conditional flex + border color derived from morale AND chronicle-hover state, canvas positioning, roster/focus toggle buttons (each ~20 lines of style object), DivergenceEmptyState banner, TileTooltip rendering. F4 migration target; the chronicle-hover color dictionary for border + box-shadow is one of the cleanest "move to SCSS @extend chain" candidates in the codebase.

**L4 — Roster + focus toggle buttons are extraction candidates.** [Lines 910-968](../src/cli/dashboard/src/components/viz/grid/LivingSwarmGrid.tsx#L910-L968) — two ~25-line button blocks with near-identical styling (border toggles color on active state, ARIA labels, position absolute). Extract `<GridTopRightButton active icon onClick label />` + use twice.

**L5 — `DivergenceEmptyState` and `TileTooltip` are extractable components.** [Lines 1004-1033](../src/cli/dashboard/src/components/viz/grid/LivingSwarmGrid.tsx#L1004-L1033) (divergence empty banner) and [lines 1034+](../src/cli/dashboard/src/components/viz/grid/LivingSwarmGrid.tsx#L1034) (tile tooltip). Each ~30-line inline JSX + styles. Extract to own files; LivingSwarmGrid shrinks by ~100 lines and each becomes independently testable.

**L6 — Game-of-Life integration has 5 refs + ~60-line seed logic.** [Lines 221-230](../src/cli/dashboard/src/components/viz/grid/LivingSwarmGrid.tsx#L221-L230) (refs) + [lines 400-460](../src/cli/dashboard/src/components/viz/grid/LivingSwarmGrid.tsx#L400) (seed/warmup/inject logic inside the main render effect). This is a self-contained sub-feature. Extract to `useGolAmbient({ snapshot, mode, eventFilter, positions, size, flares })` returning `{ gol, tick, inject }` — the main render effect just calls `drawGol(ctx, useGolAmbient(...).gol, ...)`.

**L7 — The `resolveCssColor` helper is file-local but useful elsewhere.** [Lines 115-122](../src/cli/dashboard/src/components/viz/grid/LivingSwarmGrid.tsx#L115-L122) — resolves `var(--foo)` against a computed style, falls back to the literal. Could live in `grid/cssColor.ts` and be reused by `GlyphLayer` / `FlareLayer` / `HudLayer` which currently pass the already-resolved color from here.

### Already good

- 15 helpers extracted: `computeGridPositions`, `GlyphLayer`, `FlareLayer`, `HudLayer`, `ModeOverlayLayer`, `useGridState`, `computeDeptCenters`, `GameOfLifeLayer`, `hitTest`, `ClickPopover`, `RosterDrawer`, `FeaturedSpotlight`, `useMediaQuery`, `GridSettingsDrawer`, `GridMetricsStrip`
- Ref strategy is disciplined — the 5 GoL refs are all documented with the reason they're refs vs state (avoids re-renders during decay animations)
- Reduced-motion path consistently honored across reveal, GoL warmup, and animation speed
- Palette switching via `resolveCssColor` keeps theme switching working
- Canvas HiDPI handling clean (`devicePixelRatio` with `setTransform` reset)

## Recommended phased plan

### Phase 1 — SCSS migration (low risk, high visible-cleanup)

One commit per file:

- **1a.** `SwarmViz.tsx` 34 inline styles → `SwarmViz.module.scss`. CSS custom properties on the root for leader-palette colors; fixed tokens for the rest.
- **1b.** `LivingSwarmGrid.tsx` 27 inline styles → `LivingSwarmGrid.module.scss`. Chronicle-hover color dictionary becomes `@extend` chain: `.chronicleHoverBirth`, `.chronicleHoverDeath`, `.chronicleHoverForge`, `.chronicleHoverCrisis`. Border + box-shadow pair lives in the base class.

Zero behavior change. Full 215 test suite green. Visual spot-check against the live dashboard.

### Phase 2 — Button + empty-state extraction (small, independent)

- **2a.** Extract `<GridTopRightButton>` and use twice for roster toggle + focus toggle. ~40-line win.
- **2b.** Extract `<DivergenceEmptyState>` to its own file.
- **2c.** Extract `<GridTileTooltip>` to its own file.

LivingSwarmGrid shrinks by ~150 lines with zero behavior change.

### Phase 3 — Hook extraction (medium risk, biggest structural win)

- **3a.** `useCrisisToast(state, enabled)` from SwarmViz's 80-line effect. Pure helper `findLatestUntoastedCrisis(events, seen)` gets unit-tested via node:test.
- **3b.** `useGridSettings()` co-located with `GridSettingsDrawer` absorbing the schema-versioned localStorage pattern.
- **3c.** `useLocalStorageState<T>(key, defaultValue, schemaVersion?)` as the shared primitive. Replace the pattern in both SwarmViz spots + the `gridMode` single-string persistence.

### Phase 4 — LivingSwarmGrid render refactor (biggest lift, highest payoff)

- **4a.** Extract `renderFrame()` to `renderFrame.ts`. Takes a `RenderContext` struct; returns nothing. The useEffect becomes a thin dispatcher. Pure canvas calls mean it's testable via a recording fake ctx.
- **4b.** Extract `useCanvasInteractions()` hook for pointer/click/touch handlers. Returns `{ on*, hovered, cursor, hoveredTile, popover }`. Hit-test logic moves out of component.
- **4c.** Extract `useGolAmbient()` for the Game-of-Life wiring. 5 refs + seed/warmup/inject logic collapses into one hook.

After Phase 4, LivingSwarmGrid.tsx should be ~350 lines (down from 1243) — pure wiring of hooks to DOM.

### Phase 5 — Shared concerns (cleanup)

- Move `resolveCssColor` to `grid/cssColor.ts`; dedup across layers
- Share the `ChronicleHoverPayload` type from `EventChronicle.tsx`
- Consider `<GridSlot>` wrapper for multi-leader prop delivery (F2+F3 prep)

## Out of scope

- **Performance profiling.** Render effect is ~400 lines but bounded by the drawing primitives beneath it. A flame-graph pass would tell us if any layer is the bottleneck; that's its own audit.
- **Test infra.** Current node:test pattern handles pure helpers well. Testing the render effect requires a canvas mock (jsdom's is minimal); likely a `RecordingCtx` fake rather than a real DOM-based test.
- **F2/F3 arena-ready shape.** LivingSwarmGrid is currently per-side with an `a | b` union. When N leaders land (F2+F3 spec), `side: string` + a `leaderIndex: number` generalizes. Not in F6's scope.
- **WebGL / GPU rendering.** The canvas2d pipeline is reading + drawing ~100-200 glyphs per frame + GoL grid. No evidence it's a bottleneck. Migration to WebGL is a separate project.
